**Key insight:** As companies scale, the cost driver shifts from fixed-price components (database) to usage-based components (bandwidth/egress). This inversion must be factored into pricing decisions.

## 3. Architectural Justification

### 3.1 Selected Architecture: Firebase Hosting + Client-Side Logic

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Frontend | React (Vite) | Fast build times, component reusability, rich ecosystem |
| Hosting | Firebase Hosting | Free tier covers project needs, global CDN, SSL included |
| Calculation Engine | Client-side JavaScript | Zero server costs, instant responses, no cold starts |
| Pricing Config | Static JSON file | Easy to update, no database dependency, version-controlled |

### 3.2 Why This Architecture?

1. **Cost Efficiency**: By running calculations client-side, we eliminate the need for Firebase Functions (Cloud Functions), avoiding per-invocation costs and cold start latency. For a calculator tool, server-side computation adds cost without adding value.

2. **Scalability**: Firebase Hosting serves static assets via Google's global CDN. Whether 10 or 10,000 users access CFO Bot simultaneously, hosting cost remains near-zero on Firebase's free Spark plan.

3. **Simplicity**: A single deployable unit (static React build) reduces operational complexity, deployment risk, and debugging surface area.

4. **Pricing Flexibility**: The centralized `pricing.json` approach means pricing updates require only a config change and redeploy — no code changes, no database migrations.

### 3.3 Trade-offs Acknowledged

| Decision | Trade-off | Mitigation |
|----------|-----------|------------|
| Client-side calculation | Pricing logic visible in browser | Acceptable for an estimation tool — no sensitive business logic |
| No backend API | Cannot track usage analytics | Could add Firebase Analytics if needed |
| Static pricing config | Requires redeploy for price changes | Acceptable cadence for cloud pricing updates (monthly/quarterly) |

## 5. Recommendations

1. **For startups (Scenario A):** Choose the Basic database tier and optimize compute hours. At $37.60/month, cloud costs are manageable, but the fixed DB cost dominates — evaluate whether a managed DB is necessary vs. a lightweight alternative.

2. **For scaling companies (Scenario B–C):** Focus on bandwidth optimization (caching, CDN, compression). At enterprise scale, egress becomes 75% of costs — implementing a CDN can reduce bandwidth costs by 40–60%.

3. **General guidance:** Use CFO Bot to model costs *before* committing to a cloud provider. The linear pricing models provide a reliable baseline for budgeting, and the transparent breakdown enables informed trade-off decisions.

## 6. Conclusion

CFO Bot provides deterministic, transparent cloud cost estimation based on industry-standard pricing. Our analysis shows that cost drivers shift dramatically with scale — from fixed database costs at the startup stage to variable bandwidth costs at enterprise scale. The selected Firebase architecture for hosting the bot itself validates the "serverless-first" approach: by choosing the right architecture, even the tool that estimates cloud costs can run for free.
